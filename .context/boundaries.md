# Boundaries & Ownership

What should and should not be modified, and under what conditions.

## Do Not Modify

### Frozen API (v3)
| Path | Reason |
|------|--------|
| `src/api/v3/` response shapes | External Sonarr clients depend on exact JSON structures (Overseerr, Ombi, LunaSea, nzb360, Bazarr, Tdarr). Adding fields is OK. Changing or removing fields is **never** OK. |

### Immutable Migrations
| Path | Reason |
|------|--------|
| `migrations/*.sql` (existing files) | Applied migrations are immutable — create a NEW migration to modify schema |
| `services/pir9-imdb/migrations/*.sql` (existing) | Same rule for the IMDB service database |

**Exception**: Creating new migration files is allowed and encouraged.

### Secrets & Configuration
| Path | Reason |
|------|--------|
| `.env`, `.env.local` | Contains secrets and environment-specific config |
| `config/*.toml` | User-specific configuration |
| `certs/` | TLS certificates |
| `.claude/settings.local.json` | Personal Claude Code permissions |

### Generated / Build Artifacts
| Path | Reason |
|------|--------|
| `target/` | Rust build output |
| `frontend/dist/` | Vite build output |
| `frontend/node_modules/` | NPM dependencies |
| `sbom.cdx.json` | Generated by syft, not hand-edited |

### Infrastructure (Modify With Caution)
| Path | Reason |
|------|--------|
| `Dockerfile` | Production build — test changes locally first |
| `docker-compose*.yml` | Affects production deployments |
| `.pre-commit-config.yaml` | Security pipeline — changes affect all contributors |
| `.github/workflows/` | CI/CD pipelines |

## Modify With Care

| Path | Guideline |
|------|-----------|
| `src/api/v5/` handlers | Follow existing patterns, add tests |
| `src/core/datastore/repositories.rs` | Use SQLx compile-time checked queries |
| `src/core/messaging/` | Changes affect all event subscribers |
| `src/core/scheduler.rs` | Background job changes need testing under load |
| `Cargo.toml` | Don't add dependencies without checking if existing crates cover the use case |

## Freely Modifiable

| Path | Notes |
|------|-------|
| `src/core/<domain>/services.rs` | Business logic — add tests for new logic |
| `src/core/parser/` | Parser improvements welcome — test with real release titles |
| `frontend/src/` | Follow Biome lint rules, run typecheck |
| `.context/` | Keep documentation current |
| `.claude/skills/` | Update skills as patterns evolve |
| Tests anywhere | Add and modify tests freely |

## Cross-Cutting Concerns

Changes to these areas have wide blast radius:

| Concern | Files Affected | What to Check |
|---------|---------------|---------------|
| Error handling | All handlers, services | Maintain `anyhow`/`thiserror` pattern |
| Event bus messages | `messaging/mod.rs` + all subscribers | Don't break existing event consumers |
| `AppState` fields | `src/web/mod.rs` + all handlers | Every handler receives this |
| Database models | `datastore/models.rs` + all queries | SQLx compile-time checks will catch mismatches |
| Quality definitions | `parser/quality.rs` + profiles | Affects what releases are accepted |

## Version Bumping Rules

Every commit must bump the version:
- **Main project**: `Cargo.toml` line 3
- **IMDB service**: `services/pir9-imdb/Cargo.toml` (only if IMDB code changed)
- **CLAUDE.md**: "Current version" line

## When Unsure

1. Read the file/module you're about to change first
2. Check if there's a corresponding `.claude/skills/` skill with guidance
3. For v3 API changes — don't. Add to v5 instead.
4. For schema changes — always create a new migration, never alter existing ones
